# 接入版本化 Content API，但限制管理能力的自动化边界

Blog 的 Content API 已支持草稿、管理读取、乐观并发修改、软删除、恢复和只读历史。LSForum 适配器应覆盖这些外部能力，以便以后在得到明确操作授权时无需重新设计 HTTP 集成；现有投稿采集、来源重建和改写产物不随之改变。

`run --publication auto` 的含义仍是公开发布。适配器必须显式发送 `status: published`，不能依赖服务端默认值；省略发布选择或选择 `none` 时仍不得读取或写入 Blog。成功发布必须持久化当前 `version` 与 `ETag`，供审计和未来条件修改使用。

发布前检查及未知结果确认改用经过认证的管理读取 `GET /posts/:slug?manage=true`。确认不仅比较固定 slug、标题、正文和作者，还必须从管理响应中看到明确的未软删除标记及 `published` 状态；缺少可识别的删除状态也不能当作成功。这样草稿、已删除或状态含糊的文章不会被错误认定为一次公开发布已经完成。若丢失 POST 响应且管理 GET 只返回 version 而没有 ETag，客户端保存接口规定可用于版本头的 `"<version>"` 并发令牌。

适配器可以提供 PATCH、软删除、恢复和 revisions 的窄方法，但单一端到端 Skill 当前不把它们暴露为微信字段、CLI operation、默认运行步骤或中断恢复动作。修改必须由调用方提供当前 version，客户端按部署中的 OpenAPI v1.2.0 严格发送带双引号的 `X-Post-Version`；HTTP 428 表示缺少/非法版本头，HTTP 412 是明确版本冲突，不得盲目重试或覆盖他人修改。删除只允许外部 API 定义的软删除；客户端不提供修改历史、删除历史或彻底删除能力。

这一决定把“接口适配”与“Agent 被授权执行管理动作”分开。未来若要把编辑、删除或恢复加入运营流程，需要新的显式操作语义、授权边界、持久任务模型和验收，而不能仅因适配器已有方法就自动调用。
