# Blog 草稿 API 预期契约（讨论稿）

> 状态：已被 [ADR-0009](adr/0009-separate-content-work-from-opt-in-publication.md) 取代，仅作为早期讨论记录保留。当前观察到的外部接口以 [LSForum Blog 外部接口参考](external/lsforum-blog-api-reference.md) 为仓库内参考。
>
> 本文原本是微信投稿自动化项目对草稿 API 的集成预期，不是正式接口文档，也不代表当前采用的公开发布流程。

## 集成目标

自动化系统接收一项投稿任务，生成经过内部校验的 Markdown 和静态图片，然后把它交付到指定发布目标下，形成一篇等待人工复核的 Blog 草稿。

本次自动化的完成条件是：Blog 服务已经持久接受草稿，并返回一个以后可以稳定识别该草稿的外部标识。公开发布仍由 Blog 管理后台中的人工操作完成。

## 必须满足的边界

### 1. 只创建草稿，不公开发布

自动化使用的接口必须能够创建未公开的草稿。接受请求后立即公开文章的接口不能用于这条工作流。

自动化客户端不会调用公开发布、删除文章、管理用户、部署网站等能力。即使 Blog 另有这些接口，也建议为自动化凭据提供只包含草稿和必要图片操作的最小权限。

### 2. 发布目标必须稳定且可验证

微信任务中的“目标”是一个由运营团队维护的不透明标识。Blog 侧需要提供一种确定性映射方式，将它解析为真实的作者、Blog、栏目或其他归属资源。

映射结果应使用稳定 ID，而不是只依赖作者名或组织名等自由文本。提交前应能发现目标不存在、目标停用或当前凭据无权访问，并返回可分类的错误。

实际采用配置文件映射、查询接口还是服务端别名均可，由正式接口文档决定。

### 3. 同一任务的重试必须幂等

每个本地投稿任务都有一个稳定且唯一的任务 ID。Blog 服务应接受这个 ID 作为幂等依据，或提供语义等价的机制。

- 同一个任务、相同内容重试时，应返回原草稿，不应创建第二篇草稿。
- 同一个幂等键对应不同内容时，应明确拒绝，而不是静默覆盖。
- 客户端超时但服务器已经接受请求时，后续重试或查询应能恢复原草稿结果。
- 幂等记录的有效期需要明确；对本项目而言，最好覆盖任务的完整保留期。

仅依靠自动去重 slug 不能替代任务幂等，因为标题或 slug 相同并不代表是同一个投稿任务，而同一个任务的网络重试也不应产生新 slug。

### 4. 成功必须返回稳定的草稿证据

成功响应至少应让客户端得到：

- 稳定的外部草稿 ID；
- 明确的草稿状态；
- 实际归属的目标 ID；
- 可选的后台编辑地址或预览地址；
- 能用于排障的请求或追踪 ID。

只有响应明确表示草稿已经持久化，并且外部草稿 ID 通过校验后，本地任务才会被标记为交付完成。仅返回“请求已收到”时，还需要状态查询或异步完成机制。

### 5. 支持 Markdown 与多张静态图片

正文以 Markdown 交付，文章可能包含多张按顺序出现的静态图片。需要明确：

- 图片是先上传再引用、随草稿一起上传，还是允许使用受控的远程 URL；
- 上传结果使用什么稳定资源 ID 或 URL；
- 支持的格式、单图大小、尺寸、数量和总请求大小；
- 相同图片能否按内容哈希复用；
- 正文如何引用上传后的图片，以及封面图如何指定；
- 图片处理失败时，草稿请求是否整体失败；
- 是否下载远程图片，若会下载，其超时、来源限制和安全规则是什么。

第一版自动化只处理静态图片，不要求保留 GIF 动画，也不上传或推断视频、音频内容。

## 建议提供的操作语义

最终接口路径和字段名可以不同，但完整集成通常需要以下语义能力：

| 能力 | 必要性 | 预期语义 |
| --- | --- | --- |
| 创建或恢复草稿 | 必须 | 根据稳定幂等键创建草稿，或返回之前创建的同一草稿 |
| 上传或复用图片 | 视 Blog 模型而定 | 返回可在草稿正文中稳定引用的图片资源 |
| 验证或解析发布目标 | 必须有等价方案 | 将运营侧目标映射为当前凭据可使用的稳定目标 ID |
| 查询草稿或幂等结果 | 强烈建议 | 解决响应丢失、异步接受和人工排障 |
| 更新草稿 | 可选 | 只有 Blog 确认需要且能防止覆盖人工修改时才接入 |

更新草稿不是第一版完成交付的前提。如果支持更新，需要明确版本号、条件更新或 ETag 等并发控制；自动化不能静默覆盖已经由编辑人员修改的草稿。

## 请求契约需要明确的内容

正式文档应为每个字段定义：

- JSON 类型、是否必填、是否允许 `null`、默认值；
- 字符数或字节数限制、数组上限和枚举值；
- Markdown 方言、HTML 处理、链接与图片清洗规则；
- 多语言字段的关系与缺省行为；
- 同义字段是否存在，以及同时出现时的优先级；
- 未知字段是拒绝还是忽略；
- 服务端自动生成字段的确定性规则。

为了便于客户端严格校验，同一语义最好只接受一种 JSON 类型。例如标签建议固定为字符串数组，不同时接受数组和逗号分隔字符串。

## 响应与错误契约需要明确的内容

成功与失败响应都应有完整、机器可读且版本稳定的 Schema。错误至少需要区分：

- 认证失败与权限不足；
- 发布目标不存在、停用或不可访问；
- 请求字段或 Markdown 无效，并指出具体字段；
- 幂等键冲突；
- 图片类型、大小或处理失败；
- 请求体过大或媒体类型不支持；
- 限流以及允许重试的时间；
- 临时服务故障；
- 永久业务拒绝；
- 状态未知，例如连接中断后无法确定服务端是否已经接受。

文档还需要说明哪些状态可以安全重试、是否使用 `Retry-After`、超时建议、请求追踪 ID，以及 5xx 或网关错误下的幂等保证。

## 认证、环境和运维信息

正式接入前还需要：

- 测试与生产环境的 HTTPS Base URL；
- 凭据申请、保存、轮换和撤销方式；
- 凭据 scope，以及它能访问的发布目标范围；
- 速率限制、并发限制和合理超时；
- OpenAPI 文档及其版本；
- 向后兼容、弃用和破坏性变更策略；
- 可供联调的安全测试目标，确保测试草稿不会公开出现。

凭据只会在运行时通过环境变量或操作系统凭据存储提供，不会写入微信消息、任务内容、报告或 Git。

## 对当前内部文档的主要疑问

当前材料描述的写入接口会立即发布内容，并且使用自由文本作者和组织名称；external 内容还不会关联到组织。这与本项目的草稿审核和稳定目标归属要求不同。

在讨论具体字段之前，希望 Blog 团队先确认：

1. 系统是否存在或计划提供“只创建草稿”的能力？
2. 草稿如何稳定归属到指定作者、Blog 或组织？
3. 是否能为每个本地任务提供服务端幂等保证和稳定草稿 ID？
4. 正文内多张静态图片应通过什么流程交付？
5. 自动化凭据能否限制为草稿与必要图片操作，不包含发布和删除？

如果这五点成立，剩余工作主要是字段映射、Schema 对齐和错误分类；如果只能即时公开发布，则接口能力本身与这条工作流不兼容。
